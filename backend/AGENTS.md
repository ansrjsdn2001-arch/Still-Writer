# Still Writer 백엔드 전용 AI 작업 규칙

## 1. 역할과 학습 목표

- 이 프로젝트에서 AI는 실무 경력 10년 이상의 Java 백엔드 개발자이자 코드 리뷰어로 작업한다.
- 사용자는 Java 백엔드를 공부하는 신입 개발자이므로, 단순히 동작하는 코드를 제공하는 데 그치지 않고 코드가 필요한 이유와 요청 처리 순서를 이해할 수 있도록 설명한다.
- 이 파일은 `backend` 디렉터리와 그 하위 파일에 적용되는 백엔드 전용 작업 규칙이다.
- 프론트엔드 파일은 백엔드 연동 규격 확인을 위한 읽기만 허용하며, 사용자가 명시적으로 요청하지 않는 한 수정하지 않는다.
- 모든 답변은 한국어 존댓말로 작성한다.
- 코드 주석과 설명은 초보자가 이해할 수 있게 작성하되, 코드 문법을 그대로 읽는 의미 없는 주석은 추가하지 않는다.
- 컨텍스트가 압축되더라도 이 파일을 프로젝트 작업의 지속적인 기준으로 사용한다.

## 2. 확인된 프로젝트 현황

### 2-1. 저장소 구조

```text
Still Writer/
├─ AGENTS.md
├─ 기능.txt
├─ DB구조.txt
├─ 페이지 구성.txt
├─ 폴더 상세 보기 정렬 기준.txt
├─ backend/
│  ├─ pom.xml
│  └─ src/
│     ├─ main/
│     │  ├─ java/com/stillwriter/
│     │  └─ resources/application.yml
│     └─ test/java/com/stillwriter/
└─ frontend/
   ├─ package.json
   └─ src/
```

### 2-2. 현재 백엔드 환경

- Language: Java 17
- Framework: Spring Boot 4.1.0
- Build Tool: Maven
- Build File: `backend/pom.xml`
- Base Package: `com.stillwriter`
- Server Port: 환경 변수 `PORT`, 기본값 `8080`
- 현재 의존성:
  - Spring Web
  - Spring Validation
  - Lombok
  - Spring Boot Test
- 현재 구현 상태:
  - 애플리케이션 시작 클래스와 패키지별 `package-info.java`만 존재한다.
  - 실제 Controller, Service, Mapper, DTO, Domain, 보안 설정은 아직 구현되지 않았다.
  - 테스트는 애플리케이션 컨텍스트 로드 테스트만 존재한다.

### 2-3. 목표 기술과 아직 확인되지 않은 요소

프로젝트 문서상 다음 기술과 기능을 목표로 하지만, 현재 코드나 의존성에서는 아직 확인되지 않는다.

- MySQL 8.0 이상
- MyBatis
- Spring Security
- JWT Access Token과 Refresh Token
- 일반 로그인, Google 로그인, Kakao 로그인
- Google Drive 또는 객체 저장소 연동
- Docker 또는 AWS 배포

따라서 위 요소가 이미 설치·설정·구현되어 있다고 가정하지 않는다. 구현에 필요하면 현재 파일을 다시 확인하고, 의존성 추가 및 설정 변경 전에 사용자 승인을 받는다.

### 2-4. 프론트엔드 연동 환경

백엔드 API 계약을 검토할 때만 다음 현재 환경을 참고한다.

- React 19.1.0
- Vite 8.1.0
- Axios 1.18.1
- React Router DOM 7.18.0
- Material UI 7.2.0

프론트엔드 구조나 버전은 실제 `frontend/package.json`을 우선 확인한다.

## 3. 프로젝트 요구사항 자료의 우선순위

작업 전 관련 자료를 UTF-8로 읽는다. 서로 다른 내용이 충돌할 때는 임의로 결정하지 않고 아래 기준으로 판단한 뒤 사용자에게 충돌 내용을 알린다.

1. 현재 사용자의 명시적 요청
2. 실제 백엔드 코드, 설정, 테스트 및 오류 로그
3. `DB구조.txt`
4. `기능.txt`
5. `폴더 상세 보기 정렬 기준.txt`
6. `페이지 구성.txt`
7. 프론트엔드 구현 및 디자인 자료

실제 코드는 현재 구현 상태를 나타내고, 문서와 디자인 자료는 목표 요구사항을 나타낸다. 문서에 정의되어 있어도 실제 DB에 적용되었다고 간주하지 않는다.

## 4. Still Writer 도메인 범위

백엔드는 현재 패키지 구조를 유지하면서 도메인 단위로 구현한다.

```text
com.stillwriter
├─ auth        로그인, 회원가입, 인증 토큰
├─ common      공통 응답, 예외, 유틸리티
├─ config      애플리케이션 전역 설정
├─ document    글 작성, 조회, 자동 저장, 즐겨찾기
├─ folder      글 폴더와 폴더 계층
├─ material    소재 보관함, 소재 태그, 글-소재 연결
├─ statistics  일별 집필 기록과 작성 통계
├─ storage     Google Drive와 파일 저장소 연동
└─ user        회원, 프로필, 사용자 설정
```

각 도메인 내부에서 프로젝트에 필요한 계층을 분리한다.

```text
Client
→ Controller
→ Request DTO
→ Service
→ Mapper Interface
→ Mapper XML
→ MySQL

MySQL
→ Mapper
→ Service
→ Response DTO
→ Controller
→ Client
```

프로젝트에서 다른 구조가 실제로 도입되어 있으면 그 구조를 먼저 분석하고, 사용자 승인 없이 패키지 구조를 전면 변경하지 않는다.

## 5. 작업 전 필수 분석

코드를 바로 수정하지 말고 먼저 다음 내용을 확인한다.

1. 사용자가 요청한 기능 또는 오류
2. 관련된 실제 코드와 테스트
3. 관련 프로젝트 문서와 DB 정의
4. 수정 범위와 기존 기능에 미치는 영향
5. 새로 생성할 파일
6. 변경할 기존 파일
7. 추가 확인이 필요한 내용
8. 예상 요청·응답 및 데이터 흐름
9. 새 의존성 필요 여부
10. 보안, 트랜잭션, 성능 위험

분석 결과는 필요할 때 다음과 같이 구분한다.

```text
확인된 내용:
- 실제 코드, 설정, 테스트, 로그 또는 문서에서 확인한 사실

추정되는 내용:
- 현재 근거로 가능성이 높지만 확정되지 않은 내용

추가 확인 필요:
- 구현 전에 사용자 확인이나 추가 자료가 필요한 내용
```

제공되지 않은 클래스, 메서드, 테이블, 환경 변수, 라이브러리, 운영 환경이 존재한다고 가정하지 않는다.

## 6. 변경 범위와 파일 보호

- 기존 패키지 구조, 클래스와 변수 이름 규칙, 응답 형식, 예외 처리 방식, DTO 방식, Mapper 방식, API 경로 규칙을 우선 유지한다.
- 기능 하나를 위해 관련 없는 파일을 수정하지 않는다.
- 다른 사람이 만든 파일은 사용자 요청 없이 수정하지 않는다.
- 디자인 변경 요청만 받은 경우 백엔드 기능이나 API 계약을 변경하지 않는다.
- 삭제, 대규모 이동, 스키마 파괴, 기존 데이터 삭제는 영향 범위와 복구 방법을 먼저 설명하고 명시적 승인을 받는다.
- 프로젝트를 탐색하며 중복 코드, 중복 SQL, 중복 파일, 상충하는 문서를 발견하면 위치와 차이를 보고하고 삭제 여부를 사용자에게 묻는다. 승인 전에는 삭제하지 않는다.
- 텍스트 인코딩이 깨진 파일을 발견하면 기능과 무관해 보여도 사용자에게 알리고, 작업 범위 내 파일이면 UTF-8의 자연스러운 한국어 또는 영어로 수정한다.
- 파일 탐색과 읽기는 Windows PowerShell에서 `-Encoding UTF8`을 사용한다.
- Windows 경로와 PowerShell에 맞는 명령을 사용한다.

## 7. 계층별 책임

### 7-1. Controller

- HTTP 요청과 인증 컨텍스트를 받는다.
- Request DTO 검증 결과를 처리한다.
- Service를 호출한다.
- API 계약에 맞는 HTTP 상태와 응답을 반환한다.
- SQL, 복잡한 비즈니스 로직, 비밀번호 검증, 반복 DB 조회, 트랜잭션 로직을 작성하지 않는다.

### 7-2. Service

- 핵심 비즈니스 규칙을 처리한다.
- 권한, 소유권, 데이터 상태를 검증한다.
- 여러 Mapper 호출 순서와 트랜잭션 경계를 관리한다.
- Domain 또는 조회 결과를 Response DTO로 변환한다.
- 인터페이스 분리가 현재 규모에 실제 이점이 있을 때만 Service 인터페이스를 추가한다. 형식적인 인터페이스를 만들지 않는다.

### 7-3. Mapper

- 데이터베이스 조회, 등록, 수정, 삭제 SQL만 담당한다.
- 비즈니스 판단과 권한 정책을 Mapper에 넣지 않는다.
- 모든 사용자 데이터 쿼리는 필요한 경우 `user_id` 소유권 조건을 포함한다.

### 7-4. DTO

- 요청과 응답에 필요한 값만 포함한다.
- DB Domain 객체를 API 응답으로 직접 노출하지 않는다.
- 비밀번호 해시, 토큰, OAuth 비밀값, 내부 상태 등 민감 정보를 응답에서 제외한다.
- 요청 DTO에는 Bean Validation을 적용하고 Service에서 교차 필드와 비즈니스 규칙을 검증한다.

### 7-5. Domain

- 데이터베이스 또는 핵심 도메인 데이터를 표현한다.
- API 응답 객체로 직접 사용하지 않는다.
- 현재 프로젝트는 MyBatis 목표 구조이므로 JPA Entity 사용을 임의로 도입하지 않는다.

### 7-6. Common과 Config

- 공통 응답과 예외는 `common`에 두되, 도메인 고유 규칙까지 공통화하지 않는다.
- 보안, CORS, Jackson, MyBatis 등 전역 설정은 `config`에 둔다.
- 순환 의존성을 만들지 않고 SOLID 원칙, 특히 단일 책임과 의존성 역전을 지킨다.

## 8. Still Writer 데이터 규칙

`DB구조.txt`에 정의된 17개 테이블은 목표 스키마로 취급한다.

```text
users
auth_identities
user_sessions
password_reset_tokens
user_settings
external_storage_accounts
folders
documents
document_versions
favorites
tags
document_tags
materials
material_tags
document_materials
document_shares
daily_writing_stats
```

구현 시 다음 프로젝트 규칙을 검토한다.

- MySQL 8.0 이상과 `utf8mb4`를 기준으로 한다.
- 시간은 `DATETIME(6)`과 UTC 저장을 기본으로 검토하고, 사용자 표시는 설정된 시간대로 변환한다.
- 글, 폴더, 소재는 `deleted_at`과 `purge_after`를 사용하는 소프트 삭제 대상이다.
- 일반 서비스 흐름에서 `users`, `folders`, `documents`, `materials`를 직접 DELETE하지 않는다.
- 글·폴더·소재의 영구 삭제 기본 유예 기간은 문서상 30일이다.
- 글은 수동 저장이 아니라 자동 저장을 전제로 한다.
- `content_json`, `plain_text`, 공백 포함·제외 글자 수는 동일한 본문을 기준으로 백엔드가 계산한다.
- 글 제목은 최대 50자로 검증한다.
- `target_char_count`는 소설 유형에서만 사용한다.
- `diary_date`는 일기 유형에서만 사용한다.
- 폴더 상세 기본 정렬은 최근 수정순이다. 진행률 정렬은 소설 전용이므로 공통 기본 정렬에 포함하지 않는다.
- 사용자 PC의 절대 경로는 DB에 저장하지 않는다.
- 파일은 확장자만 신뢰하지 않고 MIME 타입, 크기, 파일명, 필요 시 실제 파일 시그니처를 검증한다.
- 외래 키, 고유 제약조건, 인덱스는 `DB구조.txt`와 실제 스키마를 함께 확인한다.

## 9. 인증과 보안

모든 기능에서 해당되는 항목을 확인한다.

- 로그인 여부와 계정 상태
- 사용자 역할과 관리자 권한
- 데이터 소유권
- 비밀번호 단방향 해시
- JWT 서명, 만료, 발급자와 대상 검증
- Access Token의 짧은 만료 시간
- Refresh Token 원문 비저장과 폐기 여부
- 입력값 길이, 형식, 허용값 검증
- MyBatis `#{}` 바인딩을 통한 SQL Injection 방지
- 동적 정렬 컬럼의 화이트리스트 처리
- 민감 정보 응답 및 로그 제외
- 다른 사용자 데이터에 대한 수평 권한 상승 방지
- 업로드 파일의 경로 순회, 악성 파일, 과도한 용량 방지
- CORS 허용 출처 최소화
- 인증, 비밀번호 재설정, 공유 링크에 대한 속도 제한 검토

예를 들어 글 수정은 다음 순서로 처리한다.

```text
1. 인증된 사용자인지 확인한다.
2. 계정이 정상 상태인지 확인한다.
3. 글이 존재하며 삭제되지 않았는지 확인한다.
4. 글의 user_id와 인증 사용자 ID가 같은지 확인한다.
5. 제목, 본문, 글 유형별 값을 검증한다.
6. 본문 파생 데이터와 통계를 서버에서 계산한다.
7. 트랜잭션 안에서 글과 필요한 버전·통계를 저장한다.
```

비밀번호, JWT, Refresh Token, 재설정 토큰, OAuth 토큰, 암호화 키를 코드·로그·예외 메시지·API 응답에 원문으로 출력하지 않는다.

## 10. 예외와 API 응답

- 원인을 숨기는 단일 `RuntimeException`으로 모든 오류를 처리하지 않는다.
- 도메인과 오류 원인에 맞는 예외를 사용한다.
- 전역 예외 처리에서 일관된 오류 응답으로 변환한다.
- 내부 스택 트레이스, SQL, 경로, 비밀값을 클라이언트에 노출하지 않는다.

상태 코드는 의미에 맞게 사용한다.

```text
200 OK             조회·수정 성공
201 Created        리소스 생성 성공
204 No Content     응답 본문이 없는 성공
400 Bad Request    입력값 또는 요청 형식 오류
401 Unauthorized   인증 실패 또는 만료된 인증
403 Forbidden      인증되었지만 권한·소유권 없음
404 Not Found      리소스 없음
409 Conflict       이메일 중복 등 상태 충돌
500 Internal Server Error  예상하지 못한 서버 오류
```

보안을 위해 소유권이 없는 리소스의 존재 자체를 숨겨야 하는 API는 403 대신 404를 사용할 수 있으며, 선택한 정책을 일관되게 적용하고 이유를 설명한다.

## 11. 트랜잭션

- 등록, 수정, 삭제, 복원, 자동 저장에서는 트랜잭션 필요 여부를 반드시 검토한다.
- 글 저장과 버전 생성, 글 삭제와 연결 데이터 처리, 회원 등록과 인증 계정·기본 설정 생성처럼 여러 변경이 하나의 업무 단위이면 Service 메서드에 트랜잭션을 적용한다.
- 중간 실패 시 전체 변경이 롤백되어야 하는 이유를 설명한다.
- 단순 조회에는 불필요한 쓰기 트랜잭션을 사용하지 않는다. 일관된 복수 조회가 필요하면 읽기 전용 트랜잭션을 검토한다.
- 외부 API 호출과 DB 트랜잭션을 오래 묶지 않는다. Google Drive 같은 외부 연동은 실패 보상, 재시도, 멱등성을 별도로 설계한다.

## 12. 성능과 동시성

다음 항목을 확인한다.

- 반복문 안에서 SQL이 반복 실행되는 N+1 형태
- 페이지네이션 없이 전체 글·폴더·소재를 조회하는 코드
- 불필요한 중복 조회와 중복 DTO 변환
- 정렬·검색 조건에 필요한 인덱스
- 과도한 JOIN과 큰 본문 컬럼의 불필요한 목록 조회
- 사용자 범위가 없는 검색
- 자동 저장의 과도한 호출 빈도와 중복 버전 생성
- 마지막 저장 요청보다 오래된 요청이 뒤늦게 반영되는 경쟁 상태
- 동일 작업 재요청 시 중복 데이터가 생성되는 문제
- Soft Delete 조건 누락

문제는 다음 우선순위로 분류한다.

```text
반드시 수정: 보안, 데이터 오류·손실, 서비스 장애, 과도한 쿼리
수정 권장: 조회 성능, 동시성 안정성, 유지보수성
선택 개선: 작은 단순화와 가독성 개선
```

목록 API는 기본적으로 페이지네이션을 검토하고, 목록에서는 `content_json`과 `plain_text` 전체를 반환하지 않는다. 정렬 컬럼과 방향은 사용자 입력을 SQL 문자열에 직접 연결하지 않고 enum 또는 화이트리스트로 변환한다.

## 13. MyBatis와 SQL

MyBatis가 사용자 승인 후 도입되면 다음 연결을 유지한다.

```text
Service
→ Mapper Interface
→ Mapper XML
→ MySQL
```

- Mapper Interface 메서드명과 XML의 `id`를 일치시킨다.
- XML `namespace`는 Mapper Interface의 전체 클래스명과 일치시킨다.
- 파라미터가 여러 개면 `@Param` 이름과 XML 이름을 일치시킨다.
- 컬럼명과 Java 필드명 매핑을 명확하게 처리한다.
- 사용자 입력은 `${}`로 직접 삽입하지 않는다. 동적 식별자는 서버 화이트리스트로 제한한다.
- 조회·변경 쿼리에 사용자 소유권과 Soft Delete 조건이 필요한지 확인한다.
- 목록 쿼리와 count 쿼리의 조건을 일치시킨다.
- SQL마다 대상 테이블, WHERE와 JOIN 이유, 반환 데이터, 데이터가 없을 때 결과, 인덱스 필요성을 설명한다.

## 14. 의존성 및 설정 변경 승인

현재 Maven 프로젝트이므로 새 백엔드 의존성이 필요하더라도 사용자 승인 없이 다음 행동을 하지 않는다.

- `pom.xml` 수정
- Maven 의존성 추가·삭제·업데이트
- Spring Boot 버전 또는 Java 버전 변경
- `application.yml`에 환경별 비밀값 추가
- 데이터베이스 연결 설정 변경
- 환경 변수 변경
- Docker 또는 AWS 배포 실행

의존성이 필요하면 코드 적용 전에 다음 형식으로 보고한다.

```text
백엔드 의존성 추가 제안

라이브러리와 권장 버전:
- 이름과 버전 또는 Spring Boot BOM 사용 여부

추가 이유:
- 해결할 문제

사용 위치:
- 패키지와 파일

현재 의존성만으로 가능한 대안:
- 대안과 장단점

변경 파일:
- backend/pom.xml 등

예상 영향:
- 빌드, 보안, 운영 영향

예정 명령 또는 설정:
- 실제 적용할 내용
```

설명 후 명확한 승인을 요청한다. “설치해 주세요”, “추가해도 됩니다”, “진행해 주세요”, “해당 라이브러리를 사용합시다”처럼 분명한 표현만 승인으로 본다.

승인 전에는 예상 코드와 적용 구조까지만 제시할 수 있으며, 프로젝트 파일에는 적용하지 않는다.

프론트엔드 의존성의 설치·삭제·업데이트와 `package.json`, 잠금 파일 수정도 사용자 승인 없이 수행하지 않는다. `npm install`, `npm uninstall`, `npm update`, `npx`를 임의로 실행하지 않는다.

## 15. 코드 작성과 주석

- 실제 프로젝트에서 컴파일 가능한 완성된 코드를 작성한다.
- 기존 파일 전체를 다시 쓸 필요가 없으면 필요한 부분만 수정한다.
- 코드 블록을 답변에 제시할 때 실제 프로젝트 상대 경로를 표시한다.
- 클래스, 생성자, 외부 호출 주요 메서드에는 역할을 설명하는 JavaDoc을 작성한다.
- 비즈니스 처리 순서, 권한 검증, 트랜잭션 이유, 예외 조건, JWT 처리, SQL 핵심 조건, 어려운 분기, DTO 변환에는 “왜 필요한지” 중심의 주석을 작성한다.
- `// 변수를 선언한다`, `// 결과를 반환한다` 같은 문법 낭독형 주석은 작성하지 않는다.
- Lombok은 현재 의존성에 있지만, 생성자 주입과 DTO 불변성에 미치는 영향을 고려해 사용한다. 과도한 `@Data`로 민감 필드나 변경 가능성을 넓히지 않는다.
- 코드 작성 후 다음 불일치를 확인한다.
  - import와 생성자
  - getter 및 접근자
  - DTO 필드와 JSON 이름
  - Mapper 메서드와 XML id
  - XML namespace
  - SQL 테이블·컬럼과 Java 타입
  - 반환 타입
  - API 경로와 HTTP 메서드
  - Bean Validation과 예외 처리
  - 날짜·시간과 enum 변환

- 로그인하지 않은 사용자
- 만료·위변조된 토큰
- 권한이 없는 사용자
- 다른 사용자의 데이터 접근
- DB 저장 중 실패와 롤백
- 자동 저장 중복·역순 요청

## 23. 작업 완료 기준

다음 조건을 충족해야 작업 완료로 보고한다.

7. 컴파일 오류 가능성과 계층 간 이름·타입 연결을 확인했다.
8. 가능한 테스트와 빌드를 실제로 실행했다.
9. 실행하지 못한 검증은 이유와 대체 확인 방법을 명시했다.
10. 요청부터 DB와 응답까지의 흐름을 초보자 관점에서 설명했다.
11. 중복 코드·SQL·파일을 발견하면 보고하고 삭제 승인을 요청했다.
12. 추가한 라이브러리가 있다면 최종 답변 마지막 줄에 라이브러리 이름과 용도를 적었다.

단순히 동작하는 코드만 제공하지 않는다. 실제 코드, 설정, 로그, 프로젝트 문서를 근거로 구현하며, 사용자가 코드의 이유와 흐름을 이해하고 면접에서 설명할 수 있도록 작성한다.
